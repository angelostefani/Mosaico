Security
1.

Type: Security
File and line: mosaico/settings.py:28
Description: The SECRET_KEY has a hardcoded insecure fallback value. If the env variable is unset in any environment, Django silently uses it — sessions, CSRF tokens, and signed cookies become predictable.
Proposed fix: Remove the fallback default and raise an error (or at minimum use os.getenv('DJANGO_SECRET_KEY') with no default, letting Django fail loud).
Impact: High
2.

Type: Security
File and line: ui/views.py:19, ui/templates/ui/chat.html:84, ui/templates/ui/upload.html:77-78, ui/templates/ui/uploads.html:60-62, ui/templates/ui/collection_config.html:59-61
Description: The JWT token and the fallback FAKE_TOKEN are embedded directly into rendered HTML as JS variables. Anyone who views the page source (or exploits a caching issue) can see and reuse these tokens. If FAKE_TOKEN is a real production credential it is fully exposed.
Proposed fix: Never embed tokens in HTML. The JWT should be transmitted only via httpOnly cookies; FAKE_TOKEN should not be passed to templates at all if it carries any real authorization weight.
Impact: High
3.

Type: Security
File and line: ui/templates/ui/uploads.html:150-158, ui/templates/ui/collection_config.html:79
Description: API response data is injected directly into innerHTML without any escaping. A malicious backend response (e.g., item.upload_id containing <script>…</script>) would execute arbitrary JS in the user's browser (stored/reflected XSS).
Proposed fix: Replace innerHTML assignment with safe DOM methods (createElement, textContent) or a sanitisation step. configStatus.innerHTML should also use textContent for the message part.
Impact: High
4.

Type: Security
File and line: mosaico/settings.py:154
Description: CORS_ALLOW_ALL_ORIGINS = True is unconditional — it applies even in production (DEBUG=False). This lets any origin make credentialed cross-origin requests to the Django API endpoints.
Proposed fix: Restrict CORS_ALLOWED_ORIGINS to a whitelist (matching ALLOWED_HOSTS) and only keep CORS_ALLOW_ALL_ORIGINS = True when DEBUG=True.
Impact: Medium
5.

Type: Security
File and line: ui/templates/ui/public_chat.html:721-722
Description: public_chat.html exposes FAKE_TOKEN in JS and uses it as the Authorization: Bearer header when calling ${API_BASE}/chat directly from the browser — completely bypassing Django. An unauthenticated user can thus call the backend API as if they were authorized, using any username/collection they construct in the URL.
Proposed fix: The public-chat POST should go through the Django proxy (similar to upload), not directly to the backend. The token should never be in the client's hands.
Impact: High
Bugs

Type: Bug

Description: The chat view always uses FAKE_TOKEN for its backend call, ignoring any real JWT stored in the session. The upload and list_uploads views correctly do request.session.get('jwt_token') or FAKE_TOKEN; chat does not.
Proposed fix: Change line 171 to headers = {'Authorization': f'Bearer {request.session.get("jwt_token") or FAKE_TOKEN}'}, consistent with the other views.

7.

Type: Bug

Description: requests.post(f'{API_BASE}/chat', ...) has no timeout parameter. If the backend hangs, the Django worker thread is blocked indefinitely, potentially exhausting all workers.
Proposed fix: Add timeout=60 (or a configurable value), consistent with the upload view.
Impact: Medium
8.

Type: Bug
File and line: ui/views.py:55-62
Description: In user_login, the user is logged in via login(request, user) on line 55 before _obtain_jwt is called. If JWT acquisition fails, an error is added to the form and the login page is re-rendered — but the user is actually already authenticated. On the next page load they are silently logged in without a JWT, which breaks JWT-dependent features without any clear message.
Proposed fix: Either redirect the user home with a warning message (since Django session auth succeeded), or call logout(request) to roll back the session before re-rendering the error form.
Impact: Medium
9.

Type: Bug
File and line: ui/views.py:185-201, ui/views.py:191
Description: The list_uploads view contains server-side proxy logic (fetching /uploads from the backend) that is never actually used. The uploads.html template calls the backend API directly via JavaScript fetch. The view's proxy branch runs only when there are GET parameters, but the template never sends requests to the Django view — it always calls API_BASE directly. Additionally, line 191 uses FAKE_TOKEN instead of the session JWT.
Proposed fix: Either remove the dead server-side proxy branch and turn the view into a pure template renderer, or have the template call the Django proxy instead of the backend directly (which would also fix the token exposure).
Impact: Medium
Performance / Infrastructure
10.

Type: Bug / Infrastructure
File and line: docker/entrypoint.sh:11
Description: The Docker entrypoint starts Django with manage.py runserver — the development server. It is single-threaded, not production-hardened, and the comment in the file itself notes this. In production under any load this will be a severe bottleneck and is unsupported.
Proposed fix: Replace with gunicorn mosaico.wsgi:application --bind 0.0.0.0:${PORT:-9001} --workers 4 (and add gunicorn to requirements.txt).
Impact: High
11.

Type: Bug
File and line: ui/templates/ui/chat.html:88, ui/templates/ui/upload.html:82
Description: API_BASE is computed via window.location.origin.replace(':9001', ':9000'). This is fragile: it fails silently if the app runs on a different port or behind a reverse proxy on port 80/443 (no replacement occurs, and requests go to the wrong origin). collection_config.html and uploads.html use the correct server-side {{ api_base }} template variable.
Proposed fix: Replace the client-side port-replace with const API_BASE = "{{ settings.API_BASE|escapejs }}" (passed through the view context), making all four templates consistent.
Impact: Medium
